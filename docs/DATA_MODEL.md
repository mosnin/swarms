# Swarms — Data Model

> Status: v0.1 — implemented. Companion to [`ARCHITECTURE.md`](./ARCHITECTURE.md).
> Source of truth: the Drizzle schema under `src/lib/db/schema/`. Migrations in
> `drizzle/`.

## Conventions

- **Public IDs.** Every primary key is a stable, prefixed, non-sequential string
  (e.g. `org_…`, `skl_…`, `job_…`) generated by `src/lib/ids.ts`. Sequential
  integers are never exposed to clients.
- **Tenant scoping.** Every org-scoped table carries `organizationId`
  referencing `organizations.id`.
- **Timestamps.** Every table has `createdAt` and `updatedAt` (UTC). On
  append-only tables `updatedAt` never diverges from `createdAt`.
- **Money.** Stored as integer **minor units** (`bigint`) plus a 3-letter ISO
  currency code. No floating point.
- **Append-only tables.** `usage_ledger_entries`, `audit_events`, and
  `execution_logs` are insert-only; corrections append compensating rows.
- **Immutability.** `skill_versions` content is immutable once published.

## Domains and tables

| Domain      | Tables                                                                 |

| Identity    | `users`, `organizations`, `organization_members`, `api_keys`, `wallets`|

| Execution   | `jobs`, `job_steps`, `worker_runs`, `execution_logs`                   |

## Relationships

### Identity & tenancy

```
users ──< organization_members >── organizations
                                      │
                                      ├──< api_keys        (org service or user-owned)
                                      └──< wallets         (one per org+currency)
```

- `organization_members` is the join table (unique on `organizationId+userId`)
  with an `org_role` (`owner|admin|member`).
- `api_keys` store only a `prefix` (lookup) + `hashedKey` — never the raw key —
  plus `scopes[]`, `expiresAt`, `revokedAt`.
- `wallets` hold a cached `balanceMinor`, reconcilable from the ledger; unique on
  `(organizationId, currency)`, with a `version` column for optimistic locking.

### Catalog

```
organizations ──< skills ──< skill_versions
                     │
                     └──< skill_permissions (grants to grantee org / user)

organizations ──< connectors ──< connector_accounts   (encrypted creds only)
                       │
                       └──< connector_permissions
```

- `skills.visibility` ∈ `private | unlisted | public`. Unique `(organizationId, slug)`.
- `skill_versions.status` ∈ `draft | published | deprecated | yanked`, unique
  `(skillId, version)`. `publishedAt` is stamped on publish; **content is
  immutable afterward** (enforced by `skill-version-service`).
- `connector_accounts` never store raw secrets: only a `secretRef` (managed
  secret) and/or `encryptedCredentials` + `encryptionKeyId` (envelope-encrypted).

### Execution

```
jobs ──< job_steps
  │  ──< worker_runs ──(optional)── job_steps
  │  ──< execution_logs          (append-only)
  └── skill_versions (optional, when capabilityKind = skill)
```

- `jobs.status` ∈ `queued | running | awaiting_payment | awaiting_approval |
  succeeded | failed | cancelled`.
- Idempotency: unique `(organizationId, idempotencyKey)` ensures a paid action
  never runs or charges twice. `inputHash` detects key reuse with a different
  payload.
- `jobs` link the actor (`createdByUserId`, `apiKeyId`) and the capability
  (`skillVersionId` and/or swarm via `swarm_runs.jobId`).

### Billing

```
jobs ──< usage_ledger_entries >── wallets        (append-only, double-entry)
jobs ──< x402_payment_attempts ──< x402_payment_receipts
organizations ──< budgets
```

- `usage_ledger_entries` use `direction` (`debit|credit`) + `kind`
  (`charge|credit|refund|payment|adjustment|hold|release`); amounts are positive
  magnitudes with the sign carried by `direction`. Append-only.
- `x402_payment_attempts` are unique on `(organizationId, idempotencyKey)` and
  link to a `job` when funding a specific execution; `x402_payment_receipts`
  reference their originating attempt (`onDelete: restrict`) and the job.
- `budgets` carry `limitMinor`, `period`, `hardStop`, and a cached `spentMinor`.

### Governance

```
organizations ──< audit_events     (append-only; actorUserId / actorApiKeyId)
organizations ──< policy_rules      (effect allow|deny, priority-ordered)
```

### Swarm

```
organizations ──< swarm_templates ──< swarm_runs ──< swarm_agents
                                          │              └── skill_versions (optional)
                                          └── jobs (optional link)
```

- `swarm_runs` and `swarm_agents` reuse the `job_status` enum. Each agent meters
  its own `costMinor`; the run rolls up cost and links to a coordinating `job`.

## Append-only & immutability enforcement

These invariants are enforced at the **service layer** (the database is the
durable store; services are the only write path), with tests:

- `src/modules/catalog/skill-version-service.ts` — published skill versions
  reject content edits and re-publish. Tested in `skill-version-service.test.ts`.
- `src/modules/billing/ledger-service.ts` — the `LedgerStore` port exposes only
  insert + read; corrections append a reversing entry. Tested in
  `ledger-service.test.ts`.

(+ owner membership and a USD wallet), three skills (each with a published
`1.0.0` version), two connectors, one monthly budget, and one swarm template.
